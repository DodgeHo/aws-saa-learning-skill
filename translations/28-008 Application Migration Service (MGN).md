讲师：迁移到云时, 有两种使用情形｡

例如, 您希望从头开始,

并且希望直接利用云｡

在这种情况下, 您不需要执行迁移｡ 

但是, 如果您有内部部署的服务器和数据中心,

并且希望迁移到云, 则需要规划迁移｡

实现此目的的一种方法是使用AWS Application

Discovery Service规划迁移｡

我们的想法是, 您将扫描您的服务器, 并收集有关服务器利用率数据和互依关系映射的信息,

这些信息对您的迁移非常重要, 这样我们就可以了解如何迁移以及首先迁移哪些内容｡

因此, 您可以执行两种类型的迁移｡ 

一种称为使用连接器的无代理发现｡ 

这为您提供了有关虚拟机､ 配置､

性能历史记录（如CPU､

内存和磁盘使用情况）的信息｡

或者, 您可以运行一个代理来执行Application

Discovery Agent, 这样就可以从虚拟机中提供更多更新与更多信息.

例如, 系统配置､ 性能､ 正在运行的进程,

以及系统之间所有网络连接的详细信息, 这些信息有助于获得依赖关系映射｡

现在, 所有这些结果数据都可以在另一个名为AWS

Migration Hub的服务中查看｡

因此, 此Application

Discovery Service确实可以帮助您确定需要移动得内容以及它们是如何互连得.

但你真的需要动起来｡ 

从内部部署迁移到AWS的最简单方法是使用AWS应用迁移服务（也称为MGN）｡

这曾经被称为CloudEndure

Migration, 但现在已经被取代了｡

我们的想法是, 使用AWS应用迁移服务,

也就是MGN, 你可以进行重新托管, 也称为“提升和转移”解决方案, 在这种解决方案中,

你可以将其他云上的物理､ 虚拟或其他服务转换为在AWS上本地运行｡

这是怎么回事？

假设您有一个企业数据中心, 其中包含操作系统应用程序和数据库,

并且这些应用程序和数据库在磁盘上运行｡

接下来, 您将运行应用程序迁移服务｡

然后, 您必须在数据中心安装的复制代理将执行磁盘的连续复制,

这样您就有了低成本的EC2实例和EBS卷来获得这种数据复制｡

现在, 当您准备执行切换时, 您实际上可以从暂存迁移到生产,

并拥有所需大小的更大EC2实例,

以及与您所需性能相匹配的EBS卷｡

因此, 我们的想法是复制数据,

然后在某个时候进行切换｡

这是迄今为止最简单的方法｡ 

因此, 它支持范围广泛的平台､

操作系统和数据库｡

这可以最大限度地减少停机时间,

并降低成本, 因为您不需要雇用复杂的工程师来完成这项工作｡

此服务会自动完成此操作｡ 

原来如此｡
